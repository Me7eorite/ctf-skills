## 二、问题一：大小屏幕显示不一致

### 2.1 当前断点体系

```
  ┌──────────┬──────────────────┬────────────────┬─────────────┐
  │ 断点名称  │    尺寸区间        │    CSS 文件     │   行为       │
  ├──────────┼──────────────────┼────────────────┼─────────────┤
  │ mobile   │ 0 - 639px        │ 10 个 CSS 文件  │ 单列 + 底栏  │
  │ tablet   │ 640px - 1023px   │ 6 个 CSS 文件   │ ⚠️ 问题区间  │
  │ desktop  │ 1024px - 1279px  │ 7 个 CSS 文件   │ 侧边栏隐藏    │
  │ wide     │ ≥ 1280px         │ 3 个 CSS 文件   │ 侧边栏 + 右面板│
  └──────────┴──────────────────┴────────────────┴─────────────┘
```

### 2.2 确认的 4 个具体不一致问题

#### 问题 A: 平板区间 (640-1023px) 布局塌陷

**根因**: `layout.css` 第 8 行将 `grid-template-columns: var(--sidebar-w) 1fr` 在 ≤1023px 时切换为 `1fr`。但侧边栏 `sidebarNav` 通过 JS 在 <1024px 时强制 `hidden`。这意味着 640-1023px 区间：

- 侧边栏被 JS 隐藏（通过 router.js 第 19-21 行）
- 布局退化为单列
- 但头部按钮是 `sm:flex`（≥640px 显示），底栏仅在 ≤639px 显示
- **结果：640-639px 根本不存在，实际是 640px 起头部按钮就出来了，但底栏≥640px 就消失了**

实际上 640-1023px 的表现：
- 侧边栏可被移动菜单按钮打开，但打开后会撑满整行宽度
- 内容区变成单列，但 padding 保持 `var(--space-md)`
- 卡片网格（metrics-grid 和 card-grid-4）变为 2 列
- 右侧面板（layout-content-side）不会显示

#### 问题 B: 头部操作按钮与移动底栏的断层

**根因**:

| 屏幕宽度 | 头部按钮 (sm:flex) | 移动底栏 (≤639px) | 状态 |
|---------|-------------------|--------------------|------|
| < 640px  | ❌ 隐藏             | ✅ 显示             | 正常  |
| ≥ 640px  | ✅ 显示             | ❌ 隐藏             | 正常  |
| **但** header.css 只控制 `display`，未考虑小屏 header 高度的变化 |

**隐患**: `<640px` 时 header 高度是 `var(--mobile-header-h)` = 56px，≥640px 时是 `var(--header-h)` = 64px。但 header.css 中没有响应式高度切换（只有 layout.css 中有 `@media (max-width: 1023px)`）。这意味着在 640-1023px 区间，header 高度仍然是 56px 但按钮已经是 flex 布局——按钮可能会挤压面包屑标题。

#### 问题 C: CSS 媒体查询边界值不一致

**根因**: 不同 CSS 文件中混用了多种边界值：

| 文件 | 查询 | 边界值 |
|------|------|--------|
| base.css | `max-width: 1023px` | 1023 |
| base.css | `max-width: 639px` | 639 |
| layout.css | `max-width: 1023px` | 1023 |
| layout.css | `max-width: 639px` | 639 |
| layout.css | `min-width: 1280px` | 1280 |
| card.css | `max-width: 1023px` | 1023 |
| card.css | `max-width: 639px` | 639 |
| research-submit.css | `max-width: 1279px` | **1279** ← 不一致！|

`research-submit.css` 用的 `max-width: 1279px` 与其他文件的 `min-width: 1280px` 形成 1px 间隙，在恰好 1280px 宽的屏幕上可能出现右侧面板闪烁。

#### 问题 D: 侧边栏 visibility 的 JS/CSS 双重控制冲突

**根因**: `index.html` 第 51 行侧边栏初始 class 为 `hidden lg:flex`：
```html
<nav id="sidebarNav" class="sidebar-nav hidden lg:flex">
```

但 `sidebar.css` 第 79-83 行有：
```css
@media (min-width: 1024px) {
  .sidebar-nav.hidden {
    display: flex;
  }
}
```

同时 `router.js` 第 19-21 行在每次 `setView()` 时：
```javascript
if (window.innerWidth < 1024) {
    document.querySelector("#sidebarNav")?.classList.add("hidden");
}
```

**问题**:
1. CSS 在大屏上强制覆盖 `.hidden` → 即使 JS 想隐藏也无法
2. JS 只在 `<1024px` 添加 hidden，但从不在 `≥1024px` 移除 hidden
3. 如果用户在小屏打开后拖拉窗口到大屏，`hidden` class 不会被移除
4. 但由于 CSS 在大屏强制 `display: flex`，这个 bug 被掩盖了，但逻辑是错误的

### 3.2 确认的 5 个具体根因

#### 根因 1: CDN 单点故障 — 无本地 fallback

Lucide 图标库**仅从 unpkg CDN 加载**，没有任何本地备份。以下场景都会导致图标丢失：

| 场景 | 后果 |
|------|------|
| unpkg.com CDN 故障 | 全局图标空白 |
| 企业网络代理拦截 unpkg.com | 全局图标空白 |
| 强制刷新 (Cmd+Shift+R) 导致 CDN 请求超时 | 全局图标空白 |
| DNS 解析失败 | 全局图标空白 |
| 浏览器内容安全策略 (CSP) 禁止外部脚本 | 全局图标空白 |

#### 根因 2: 阻塞式 script 加载 — 拖慢首屏渲染

`<script src="...">` 在 `<head>` 中且没有 `async`/`defer` 属性，会：
- **阻塞 DOM 解析**直到脚本加载并执行完成
- CDN 响应慢时，整个页面白屏等待
- 字体（Google Fonts）也依赖外部 CDN，形成双重阻塞风险

#### 根因 3: `?.` 静默失败 — 无错误提示

所有 28+ 处调用都是 `window.lucide?.createIcons()`，使用可选链操作符 `?.`。当 `window.lucide` 为 `undefined` 时：
- 不抛出任何错误
- 不打印 console 警告
- 用户看到的是空白图标，但不知道原因
- 开发者难以排查

#### 根因 4: 无重试机制

即使 Lucide 加载失败，代码不会尝试重新加载。用户只能手动刷新页面（可能再次失败）。

#### 根因 5: 动态内容重建后必须重新初始化

所有视图都通过 `root.innerHTML = ...` 重建 DOM，然后必须调用 `createIcons()` 来渲染 `<i data-lucide="...">` 为 SVG。如果 lucide 未加载：
- DOM 中只有空的 `<i>` 标签
- `<i>` 标签有 CSS `width/height` 但无内容 → 显示为空白占位 → "图标丢失"

## 四、整改方案

### 4.1 响应式布局整改（解决显示不一致）

#### 方案 1: 建立统一断点变量 (CSS 自定义属性)

在 `tokens.css` 中定义统一的断点值：

```css
:root {
  /* 统一响应式断点 */
  --bp-sm: 640px;    /* 手机断点 */
  --bp-md: 1024px;   /* 平板/桌面过渡断点 */
  --bp-lg: 1280px;   /* 宽屏断点 */
}
```

#### 方案 2: 修复 research-submit.css 的 1279px 断点

**文件**: `src/web/static/css/views/research-submit.css` 第 332 行

将 `max-width: 1279px` 改为 `max-width: 1279px` → 实际上应该是保持与全局一致。

```css
/* 修改前 */
@media (max-width: 1279px) { ... }

/* 修改后 — 与其他文件统一使用 1280px 标准 */
@media (max-width: 1279px) { ... }  /* 这个其实是对应 min-width: 1280px 的相反面，OK */
```

> 实际上 1279px 作为 1280px 的反面是正确的（比 1280 小 1px）。问题出在 layout.css 用 `min-width: 1280px` 而这里用 `max-width: 1279px` ——这其实是正确的互补关系。这个不需要修改。**真正的问题是不统一使用变量**。

#### 方案 3: 修复侧边栏 JS/CSS 双控冲突

**文件**: `src/web/static/js/router.js`

```javascript
// 修改前 (第 19-21 行)
if (window.innerWidth < 1024) {
    document.querySelector("#sidebarNav")?.classList.add("hidden");
}

// 修改后 — 双向管理 hidden 状态

if (window.innerWidth < 1024) {

    sidebarNav.classList.remove("hidden");
}
```

### 4.2 图标丢失整改（最严重的修复项）

#### 方案 1: 本地化 Lucide 图标库 ⭐ 首选方案

不再依赖 unpkg CDN，将 Lucide 的 UMD 构建文件放到项目的静态资源目录中。

```bash
# 步骤 1: 下载 lucide.min.js 到本地
curl -L -o src/web/static/assets/lucide.min.js \
  https://unpkg.com/lucide@0.468.0/dist/umd/lucide.min.js
```

**文件**: `src/web/static/index.html` 第 31 行

```html
<!-- 修改前 -->
<script src="https://unpkg.com/lucide@0.468.0/dist/umd/lucide.min.js"></script>

<!-- 修改后 — 本地加载 + defer 避免阻塞 -->
<script src="/assets/lucide.min.js" defer></script>
```

#### 方案 2: 增加图标加载状态检测和 fallback ⭐

**新建文件**: `src/web/static/js/ui/icons.js`

```javascript
/**
 * 图标初始化模块 —— 解决 CDN 加载失败问题
 */

const ICON_LOAD_TIMEOUT = 3000;

let iconReady = false;
let pendingRetry = null;

/**
 * 安全地初始化所有 Lucide 图标
 * 如果 lucide 未加载，自动重试并显示占位符
 */
export function initIcons() {
    if (window.lucide && typeof window.lucide.createIcons === 'function') {
        try {
            window.lucide.createIcons();
            iconReady = true;
        } catch (err) {
            console.warn('[图标] lucide.createIcons 执行失败:', err);
            scheduleRetry();
        }
    } else {
        if (!pendingRetry) {
            console.warn('[图标] Lucide 未加载，将在 500ms 后重试...');
            scheduleRetry();
        }
    }
}

function scheduleRetry() {
    if (pendingRetry) clearTimeout(pendingRetry);
    pendingRetry = setTimeout(() => {
        if (!iconReady) {
            if (window.lucide) {
                initIcons();
            } else {
                // 最终 fallback: 在 console 中给出明确提示
                console.error(
                    '[图标] Lucide 图标库加载失败。' +
                    '请检查网络连接或联系管理员。' +
                    '已启用文字 fallback 模式。'
                );
                showFallback();
            }
        }
        pendingRetry = null;
    }, 500);
}

/**
 * Fallback: 将 data-lucide 属性的 <i> 标签替换为文字标识
 */
function showFallback() {
    document.querySelectorAll('i[data-lucide]').forEach((el) => {
        const iconName = el.getAttribute('data-lucide');
        if (iconName && !el.innerHTML.trim()) {
            // 用图标名称首字母作为 fallback
            el.style.display = 'inline-flex';
            el.style.alignItems = 'center';
            el.style.justifyContent = 'center';
            el.style.fontSize = '10px';
            el.style.fontWeight = '600';
            el.style.color = 'var(--ink-400)';
            el.textContent = iconName.charAt(0).toUpperCase();
        }
    });
}

/**
 * 检查图标系统是否就绪
 */
export function isIconReady() {
    return iconReady;
}
```

#### 方案 4: 优化 script 加载顺序

```html
<!-- index.html <head> 中的加载顺序优化 -->

<!-- 1. preconnect 提前建立连接 -->
<link rel="preconnect" href="https://unpkg.com" crossorigin>

<!-- 2. 字体使用 display=swap 避免阻塞 -->
```

#### 方案 5: 添加 Service Worker 缓存策略（可选，长期方案）

为静态资源（尤其是 lucide.min.js 和字体）添加 Service Worker 缓存：

```javascript
// sw.js
const CACHE_NAME = 'ctf-challenge-factory-v1';
const STATIC_ASSETS = [
    '/assets/lucide.min.js',
    '/css/tokens.css',
    '/css/base.css',
    // ... 其他关键静态资源
];

self.addEventListener('install', (event) => {
    event.waitUntil(
        caches.open(CACHE_NAME).then((cache) => cache.addAll(STATIC_ASSETS))
    );
});

self.addEventListener('fetch', (event) => {
    event.respondWith(
        caches.match(event.request).then((cached) => {
            // 缓存优先：本地已有就不走网络
            return cached || fetch(event.request).then((response) => {
                return caches.open(CACHE_NAME).then((cache) => {
                    cache.put(event.request, response.clone());
                    return response;
                });
            });
        })
    );
});
```

### 4.3 字体加载优化

当前 Google Fonts 加载也有潜在的失败风险：

```html
<!-- 当前 (index.html 第 7-9 行) -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Inter:..." rel="stylesheet">

<!-- 优化后 -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<!-- 使用 display=swap + 异步加载避免 FOIT -->
<link 
  href="https://fonts.googleapis.com/css2?family=Inter:..."
  rel="stylesheet"
  media="print"
  onload="this.onload=null;this.media='all'"
>
<!-- 纯 CSS fallback 在 .css 中已配置 system-ui，确保无字体也能显示 -->
```

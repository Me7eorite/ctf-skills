## Why

`add-postgres-persistence` landed the PostgreSQL substrate and an empty Alembic baseline. Operators still cannot say "research SQL injection coverage" and get persisted sources + findings — every matrix row is hand-filled. This change introduces the first real tables and the first end-to-end research workflow so plan generation, evaluation, approval, and the eventual UI have something to stand on.

## What Changes

- New Alembic revision `0002_research_tables` introduces eight tables and three enums:
  - `challenge_categories` lookup table (`code` primary key, `display_name`, `description`). Seeded by the same revision with the three currently-supported categories (`web`, `pwn`, `re`) so the FK constraint has something to point at. Future categories (`crypto`, `ai`, `misc`, `iot`) are added with a single SQL `INSERT` — no schema migration required.
  - `agent_roles` lookup table (`code` primary key, `display_name`, `description`). Seeded with `research` (`研究 Agent`); `planning` and `shard_execution` rows arrive in their respective changes. Same extensibility pattern as `challenge_categories`.
  - `hermes_profile_bindings` (`role` text fk → `agent_roles.code`, `profile_name` text, `description`, `status` text CHECK enabled|disabled, `last_used_at`, `last_used_run_id` nullable fk → `research_runs.id`, timestamps). Binds each project-internal agent role to a Hermes profile name (the directory under `~/.hermes/profiles/<profile_name>/`). We do NOT store profile contents (SOUL.md, config.yaml, skills, sessions, memory) — those stay owned by Hermes. We only persist the binding and forensic metadata.
  - `generation_requests` (`category` text fk → `challenge_categories.code`, topic, target_count, difficulty_distribution jsonb, runtime_constraints jsonb, `seed_urls jsonb`, `max_attempts int default 3`, status enum, timestamps). Category is required; the FK is the DB-enforced whitelist so research output cannot mix categories by accident. `seed_urls` persists the operator-supplied prompt inputs so workers can execute later. `max_attempts` caps how many `research_runs` the system will create per request before flipping the request to `failed`.
  - `research_runs` (fk → generation_requests, `parent_run_id uuid` nullable fk → research_runs (retry chain), `attempt int default 1`, status enum, `claimed_by text`, `claim_token uuid`, `claimed_at`, `heartbeat_at`, `lease_expires_at` (queue control columns, all NULL while `queued`), started_at, finished_at, `last_error text`, hermes_log_path, **profile_name_used** — denormalized snapshot of which Hermes profile was actually invoked, written once at execution and never updated; provides forensic traceability).
  - `research_sources` (fk → research_runs, url, title, summary, content_hash indexed, fetched_at, raw_text_path).
  - `research_findings` (fk → research_runs, kind enum, label, summary).
  - `research_finding_sources` join table; every finding MUST reference ≥ 1 source.
- Add `src/domain/research.py` with dataclass DTOs and validators (difficulty distribution sums to target_count; finding-without-source rejected).
- Add `src/persistence/models/research.py` (SQLAlchemy 2 declarative) and `src/persistence/repositories/research.py` (typed CRUD + queries, no business logic).
- Add a new `src/services/` package with `ResearchJobService`, `ResearchAgentExecutor`, and `ResearchWorker`. Queue state changes use short transactions; Hermes invocation happens outside any database transaction.
- Add `prompts/research_prompt.md` defining the Research Agent contract: inputs (topic, count, distribution, optional operator seed URLs), output (single JSON on stdout with `sources[]` and `findings[]`; every finding declares `source_indices: int[]`).
- Extract reusable Hermes subprocess plumbing from `hermes/runner.py` into `hermes/process.py` and add `hermes/research.py::invoke_research_agent`. The invocation always passes `-p <profile_name>` resolved from `hermes_profile_bindings` at execution time; falls back to the `default` profile (and logs a WARNING) if the binding is missing or `disabled`. `HermesRunner` stays narrow (shard execution). Hermes never touches the database.
- Add `src/services/research_job_service.py::ResearchJobService` (short-transaction queue operations: `claim_next_run`, `heartbeat`, `mark_run_completed`, `mark_run_failed`, and `complete_run_with_results` — one short transaction that persists sources/findings and writes the terminal `completed` status, all token-fenced and raising `StaleClaimError` when the claim is no longer ours) and `src/services/research_agent_executor.py::ResearchAgentExecutor` (in-process orchestration: claimed run → spawn heartbeat thread → invoke Hermes outside any transaction → parse → call short-transaction handlers, swallowing `StaleClaimError` cleanly). Splitting them keeps DB ops isolated from subprocess I/O.
- Add CLI subcommand group `challenge-factory profile {list,show,bind,enable,disable,hermes-available}` for operator management of role→profile bindings. `bind` verifies the profile exists in Hermes (`hermes profile show <name>`) before persisting; `hermes-available` shells out to `hermes profile list --json` to show what is currently installed.
- `challenge-factory research submit` is asynchronous: it INSERTs `generation_requests` + an initial `research_runs` (status=queued) and returns immediately. Add `challenge-factory research worker --agent-id <W> --loop [--max-jobs N] [--poll-interval-seconds S] [--lease-seconds T]` as the long-running execution process. Multiple workers may run concurrently on the same or different hosts and coordinate via `SELECT ... FOR UPDATE SKIP LOCKED` plus a per-claim `claim_token` fencing token. Add `challenge-factory research wait <run_id> [--timeout S]` for operators who want to block until a specific run reaches a terminal status.
- Add read-only HTTP endpoints `GET /api/research/requests`, `GET /api/research/requests/{id}`, `GET /api/research/categories`, `GET /api/research/runs?status=...&claimed_by=...`, `GET /api/research/queue/stats` (counts by status, oldest queued age, runs near lease expiry), `GET /api/profile/bindings`, `GET /api/profile/bindings/{role}`. Existing dashboard HTML/JS is untouched.
- Raw fetched page text and Hermes log files live on disk under `work/research/sources/` and `work/research/logs/`. PostgreSQL holds metadata + content_hash only.
- Extend `tests/app/test_dependency_direction.py` with `services` and rejection scenarios (`hermes → services`, `services → web`, `domain → services`).
- Operator supplies seed URLs via `--seed-url`. Automated web crawling is out of scope for this change.

**BREAKING**: None. No existing module imports the new tables or the new `services` package.

## Capabilities

### New Capabilities
- `research-planning`: generation request lifecycle (scoped to one challenge category via a `challenge_categories` lookup table, with `max_attempts` retry cap), research run lifecycle as a PG-backed job queue (SKIP LOCKED claim, lease + heartbeat, retry via new rows chained by `parent_run_id`), source + finding model with the "every finding references ≥ 1 source" invariant, the no-direct-shard-promotion rule, the Hermes Research Agent prompt contract, and forensic recording of `profile_name_used` on each run. Adding a new category at runtime is a single `INSERT INTO challenge_categories` — schema does not need to change. Multiple workers may run concurrently and recover automatically from each other's failures via lazy lease expiration.
- `hermes-profile-binding`: maps project-internal agent roles (`research`, later `planning`, possibly `shard_execution`) to Hermes profile names. Operators read and modify bindings via CLI; future UI in change 4. Profile contents themselves (SOUL.md, config.yaml, skills, sessions, memory) remain owned by Hermes and are NOT mirrored into our database.

### Modified Capabilities
- `module-architecture`: add `services` to the recognized packages list and the dependency direction matrix. New allowed edges `cli → services`, `web → services`, `services → persistence`, `services → hermes`, `services → domain`, `services → core`. Forbidden edges `hermes → services`, `domain → services`, `services → web` remain forbidden and are tested.

## Impact

- Adds CLI subcommand group in `src/cli.py`. Adds HTTP read endpoints in `src/web/server.py` and a thin read model.
- Adds tests under `tests/app/`: `test_research_domain.py`, `test_research_repository.py` (postgres-marked), `test_research_executor.py` (postgres-marked, Hermes mocked), `test_research_worker.py` (postgres-marked), `test_research_queue_concurrency.py` (postgres-marked), `test_research_lease.py` (postgres-marked), `test_research_prompt.py`, `test_research_cli.py`, `test_profile_cli.py`, `test_research_alembic.py` (postgres-marked). - Filesystem: creates `work/research/sources/` and `work/research/logs/` lazily on first run. Out of scope: object-storage backend.
- Dev DB `challenge_factory` on 192.168.6.150 already exists from the previous change; the new revision lands on top of `0001_baseline`. Repository and Alembic tests reuse `challenge_factory_test`.
- `src/core/state.py`, `work/state.sqlite3`, the shard queue, `work/shards/`, and any prompt other than the new `research_prompt.md` are NOT touched. The event SQLite store remains the source of truth for shard progress; PostgreSQL covers research metadata only.

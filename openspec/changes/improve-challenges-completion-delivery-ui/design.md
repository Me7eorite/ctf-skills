## Context

完成题目页当前已经有整体交付包下载，但单题交付缺口仍然存在。页面上的筛选、统计和状态标签曾经偏向“流程展示”，现在更需要服务于完成后的交付动作与排查动作。

后端已有交付打包器 `Packer`，其核心能力是扫描已通过构建的题目并生成交付格式 v2。单题下载不需要新打包格式，只需要在现有打包结果上做范围收敛，并把下载动作暴露给前端。

## Goals / Non-Goals

**Goals:**
- 默认进入完成题目页时稳定停留在“全部”筛选。
- 页面上展示更直接的完成数量、可交付数量和单题操作入口。
- 支持单个完成题目的交付下载，并尽量复用现有打包逻辑。
- 保留整体交付包下载能力。

**Non-Goals:**
- 不改变交付包格式本身。
- 不重做完成题目背后的判定规则。
- 不引入新的前端路由或新的下载协议。

## Decisions

1. **单题下载复用现有 `Packer`，通过题目 ID 收敛选择范围**
   - 这样可以沿用现有 zip/pdf/xlsx 产物的生成逻辑，避免维护两套打包实现。
   - 备选方案是在 `Packer` 外层手工拼单题 zip，但会复制大量目录约定和命名规则，风险更高。

2. **前端在题目行/卡片内提供单题下载按钮**
   - 下载动作与题目本身绑定，符合用户“看到即下载”的心智。
   - 备选方案是在页头统一放一个下拉菜单，但会让单题入口离题目信息过远。

3. **默认筛选在进入视图时显式重置为 `all`**
   - 仅靠初始化值不够稳，因为跨视图切换会保留内存状态。
   - 备选方案是依赖刷新页面重置，但不满足单页应用的使用习惯。

4. **统计展示以“完成题目 / 可交付题目 / 单题可下载”作为主信息**
   - 这样能避免“第一次/未知/运行中”等流程状态干扰完成页的核心任务。
   - 备选方案是继续强调构建与执行细节，但会削弱页面目的。

## Risks / Trade-offs

- [Risk] 单题下载与整体交付包共用接口时，参数约束不严可能误打包多个题目。→ 在后端明确单题选择条件，并对结果数量做校验。
- [Risk] 页面内新增按钮会让表格更拥挤。→ 在桌面端保留行内操作，在移动端收敛为卡片尾部操作。
- [Risk] 默认重置筛选可能覆盖用户返回页面时的偏好。→ 只在进入完成题目视图时重置，不影响其他页面。
